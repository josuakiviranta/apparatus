## Core Idea

`pipelines/smoke/` at the repo root is a dead fixture directory hiding in plain sight. CONTEXT.md (the §Smoke-pipeline-scenario glossary entry, after the 2026-05-04 rename) declares the canonical home as `<repo>/.apparat/scenarios/<name>/`, and twenty-plus scenarios already live there driven by per-folder vitest tests. But `README.md:222` still tells contributors that smoke fixtures live under `pipelines/smoke/`, so the parallel-implement chunk-2 work wrote a fresh `parallel-illumination-to-implementation` fixture there — five files of orphaned scaffolding whose own plan (Task 2.6 findings, IMPLEMENTATION_PLAN.md:1177-1182) admits "no Vitest test in this repo iterates `pipelines/smoke/*.dot`."

## Why It Matters

The drift is not theoretical:

- `pipelines/smoke/parallel-illumination-to-implementation.dot` plus `parallel-illumination-to-implementation/{setup-fixture.sh,batch-orchestrator-stub.sh,capture-pre-sha.sh,plan.md}` exist at the wrong tier. No `*.test.ts` references them; `grep "pipelines/smoke" src/cli/tests/` returns empty. The fixture is operator-driven manual smoke — i.e. unreachable unless somebody remembers it.
- The same parallel-implement pipeline already has 22+ siblings under `.apparat/scenarios/` (e.g. `chat-end-to-end`, `static-multi-node`, `scheduler-shape-collision`) that ARE exercised by per-folder tests like `src/cli/tests/scheduler-shape-collision-scenario.test.ts` and `pipeline-illum-to-impl-*-folder.test.ts`. The two-home split has zero technical justification.
- The lens "scenario tests catch what unit tests miss" only pays out when scenarios actually run. A scenario in a directory no runner watches is documentation that rots without warning — strictly worse than no scenario at all, because it looks like coverage.
- The README is the entry point for new contributors and for the meditator agent's own orientation reads. Keeping line 222 means future chunks will keep planting fixtures under `pipelines/smoke/`, compounding the drift.
- Strategic angle: this is the same pattern as the `.apparat/.apparat/` ghost and the `.mcp-meditate-*.json` orphans flagged in `2026-05-13T0736-meditate-no-project-orientation-and-mcp-orphans.md` — solo-developer tooling silently grows two homes for the same artefact whenever a rename happens without a sweep. The illumination `2026-05-13T0805-scratch-sediment-needs-an-apparat-sweep-command.md` calls for a sweep command; this is one more candidate row in that sweep, with a *concrete* migration target.

## Revised Implementation Steps

1. **Move the fixture.** `git mv pipelines/smoke/parallel-illumination-to-implementation.dot .apparat/scenarios/parallel-illumination-to-implementation/pipeline.dot` and `git mv pipelines/smoke/parallel-illumination-to-implementation/* .apparat/scenarios/parallel-illumination-to-implementation/` (folder name matches the dot file's basename per the `.apparat/scenarios/<name>/pipeline.dot` convention). Also move the orphan sibling agent file `pipelines/smoke/plan-scheduler.md` into the same folder so the agents-next-to-pipeline rule continues to hold.
2. **Adjust `script_file=` paths inside the moved `pipeline.dot`.** They were written assuming the dot lives at `pipelines/smoke/` and resolve siblings via `parallel-illumination-to-implementation/<script>.sh`. After the move the siblings are co-located, so each `script_file=` becomes just `<script>.sh` (matches the existing `.apparat/scenarios/*` shape).
3. **Add a vitest runner** at `src/cli/tests/pipeline-parallel-illum-to-impl-smoke-folder.test.ts` patterned after `scheduler-shape-collision-scenario.test.ts` — validate + dry-run the moved `pipeline.dot`, asserting it parses and the stub orchestrator returns the expected `orchestrator_result`. This is the missing piece that turns "operator-driven manual smoke" into automated coverage and makes the scenario-tests-catch-what-unit-tests-miss lens actually pay out.
4. **Excise the old location.** `git rm -r pipelines/smoke/` (the directory empties after step 1). Confirm nothing else references it: `rg "pipelines/smoke"` should return only README/IMPLEMENTATION_PLAN/sessions/triage hits — the README one gets fixed in step 5, the rest are historical and untouched per ADR-style "append-only" convention.
