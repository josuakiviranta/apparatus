## Core Idea

Today's commit (1921f08) replaced 15 deterministic vitest smokes with a 150-line prose Phase 2 inside `.apparat/pipelines/illumination-to-implementation/tmux-tester.md`. That prose now interleaves two kinds of work: pure mechanics (`ls -d` discovery, self-skip predicate, `apparat pipeline validate`, parse required `--var` keys, append per-row entry, roll up counts) and LLM judgment (relevance reasoning over the diff, agent-as-human prompt answers, hang/glitch detection, fix-step triage). The mechanics half does not need an LLM; concentrating it inside the prompt makes the verification surface a 30 KB string the model has to obey, on a path that the deletion just made the *only* check for `.apparat/scenarios/*`.

## Why It Matters

The deletion was the right call — `static-multi-node` proved structural smokes were coverage theatre — but it concentrates risk:

- **Single-prompt drift.** Phase 2 procedure (`tmux-tester.md:223-303`) is the only thing standing between scenario rot and merge. The prompt-as-program lens (`prompt-as-program-philosophy.md`) names exactly this trade-off: trust depends on the LLM obeying a long document, and attention to early sections drifts as context grows. Today's procedure block grew from one paragraph to ~150 lines in one commit; nothing structurally bounds further growth.
- **Asymmetric verification across tiers.** The bundled tier (`src/cli/pipelines/`) keeps a deterministic live-shape check via `bundled-pipelines-self-sufficient.test.ts` (53 LOC, vitest-discovered). Project-local scenarios (`.apparat/scenarios/`, 17 entries) now have *no* on-build signal — they are exercised only when `illumination-to-implementation` lands a real implementation AND Phase 2's diff-relevance filter pulls them into INCLUDE. Two tiers, both "pipelines" per ADR-0007/0008, with verification surfaces that no longer rhyme.
- **Mechanics in prose lose locality.** Today, "list scenario folders" lives in a bash heredoc in `tmux-tester.md`. Tomorrow, when slug or scenario-folder conventions evolve (the 2026-05-08 spec already flagged the hyphen/underscore convention as a `review_gate` recommendation), every change has to thread an LLM-readable prose update instead of a code edit + test. Locality drops, change cost rises.
- **Vision check.** The project's vision (`VISION.md`) frames apparat as orchestration of agents, "agent doing one job". Phase 2 today is one agent doing six jobs. A `discover_scenarios` tool node upstream of tmux-tester would restore the "one job" shape: tmux-tester drives runs and judges, the tool produces the worklist.

## Revised Implementation Steps

1. **Inventory Phase 2 sub-steps** in `.apparat/pipelines/illumination-to-implementation/tmux-tester.md:223-303`. For each line, label MECHANICS or JUDGMENT. Expected partition: discovery, self-skip, validate-first, parse-required-vars, command construction, per-row append → MECHANICS. Relevance reasoning, agent-as-human answer choice, hang/glitch detection, fix-step trigger → JUDGMENT.
2. **Extract MECHANICS into a `discover_scenarios` tool node** upstream of tmux-tester. Add to `pipeline.dot` as `type="tool"`, `script_file="discover-scenarios.mjs"`, `outputs: scenarios` (array of `{folder, vars, validate_status, validate_error}`). Wire `tmux_tester [inputs: ..., discover_scenarios.scenarios]`.
3. **Shrink Phase 2 prose** to the JUDGMENT half — relevance INCLUDE/SKIP per row, agent-as-human plausible-answer templates, hang detection, observation criteria, per-row reason logging. The aggregation contract for `### Scenarios run` stays prose because it's user-facing render shape, not mechanics.
4. **Add a smoke check** in `discover-scenarios.mjs`: if any scenario fails `validateGraph` at discovery time, emit it as a FAIL row directly without ever entering the run loop. This pulls structural-class bugs (slug mismatch, missing agent file) up to the deterministic tier — which is what the deleted smokes did, but without the 15-files-of-cargo-cult shape. Frame this in the implementing PR as a tier-rebalance, not a re-introduction of bundled scenario tests, and surface it at `review_gate` for explicit accept/decline (the user previously declined "bundled scenario tests" — this is a different shape, but the boundary is theirs to draw).
5. **Add a reading note in `CONTEXT.md`** under the scenarios entry: "verification of `.apparat/scenarios/` runs only inside `illumination-to-implementation` Phase 2; if you change scenarios without running an illumination, drive `apparat pipeline run .apparat/scenarios/<name>/pipeline.dot --project .` by hand to confirm." This is a stop-gap until step 2 lands; cheap to revert when the tool node exists.
6. **Cap Phase 2 line count** with a CI-side soft guard or review-checklist note: if `tmux-tester.md` Phase 2 prose exceeds N lines, that is a signal to extract more mechanics. The exact N is judgment; the discipline is what matters — the prose surface does not grow unbounded just because the LLM can read it.
