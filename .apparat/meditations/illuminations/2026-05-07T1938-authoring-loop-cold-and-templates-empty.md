## Core Idea

The prior illumination (`2026-05-07T1907-pipeline-mission-control-fragmentation.md`) covered the **R** in pipeline CRUD — list, show, trace, runs, replay. The other three letters are missing. There is no `pipeline create`, no `pipeline fork`, no `pipeline edit / watch`, no `pipeline rm`. `dist/templates/.gitkeep` is the lone fossil of the chunk-5 / chunk-6d "templates infra + pipeline-create / pipeline-refine" work that MEMORY.md says shipped — it didn't, or it got reverted. Authoring a pipeline today is: `mkdir`, hand-write `.dot`, hand-write each sibling `.md`, fire `pipeline validate` against the void, fix one error, run it again, repeat. The validator is single-shot; the loop is cold. Meanwhile the apparat philosophy — pipelines are the web; gates interview humans — is exactly the seam an authoring UX should reuse. Apparat does not yet author apparat pipelines.

## Why It Matters

Concrete cold spots in the human-author loop:

- **Create**: `src/cli/program.ts` registers `run`, `validate`, `list`, `trace`, `show` only. `src/cli/commands/pipeline/list.ts:23,28` advertises `apparat pipeline create <name>` — false promise (already noted in the prior illumination, not yet acted on). `dist/templates/.gitkeep` is the only file under `dist/templates/`; the chunk-5 / chunk-6d templates infra (per MEMORY.md `2026-04-27-chunk-5-shipped.md`, `2026-04-28-chunk-6d-shipped.md`) is either reverted, never wired, or living in some untracked corner. Either way, the user who follows the hint hits a brick wall.
- **Update**: `apparat pipeline validate <name>` is the only feedback channel for a pipeline-in-progress. There is no file-watcher, no re-validate-on-save, no diff-since-last-clean. A human (or LLM author) types one attribute, runs validate, reads diagnostics, types again. Each cycle reloads the parser, the validator, the resolver — a deep module *under* `loadPipeline()` (`src/cli/commands/pipeline-invocation.ts`) that should be cheap to call repeatedly but isn't surfaced as a long-lived watcher.
- **Fork**: the two-tier read (bundled `src/cli/pipelines/<name>/` vs. project-local `<project>/.apparat/pipelines/<name>/`) implicitly invites *"I want my own version of janitor"*. There's no copy primitive. The user reads the bundled folder, learns its shape, then writes their version by hand or by `cp -R` against an `npm root -g` path the SKILL.md tells them to compute themselves. That's a shallow authoring path the engine already knows the answer to.
- **Delete**: pipelines are folders the user removes with `rm -rf`, but the daemon (`src/daemon/state.ts`) may still hold a `heartbeat pipeline <name>` entry pointing at the now-missing folder. CRUD-as-checklist: `D` is missing, and the missing `D` cross-cuts a different module's state.

Strategic compass — VISION.md says a working pipeline should "feel like delegating to someone who already understands the shape of the problem." For authoring, that means the CLI shouldn't make the human cold-start a `.dot` from a man-page. Apparat already owns the answer (gates + agents + the validator), and yet apparat asks the human to assemble those answers into a `.dot` outside apparat. The deep-module move (`stimuli/deep-modules-hide-complexity.md`) is to fold the three feedback surfaces (README anatomy, `program.ts addHelpText`, `pipelines.md`) **and** the missing CRUD verbs into one seam: a bundled pipeline that authors pipelines. Apparat authoring apparat is the same loop as `meditate` — a gate-driven interview that ends with a validated artefact on disk.

This is also a `crud-is-a-checklist-not-a-menu` reading: pipelines are a first-class persisted resource. They need C, R, U, D — not the present buffet of "we shipped R, partially."

## Revised Implementation Steps

1. **Audit `dist/templates/.gitkeep` and chunk-5 / chunk-6d's claimed shims.** Cross-reference MEMORY.md `2026-04-27-chunk-5-shipped.md` against `git log --diff-filter=D -- dist/templates/`. Either restore the templates folder + `pipeline create` shim or delete the placeholder and the false hint in `pipeline/list.ts:23,28`. Stop shipping half-landed infra.
2. **Add `apparat pipeline fork <bundled-name> [--project x]`**. Copies `<npmRoot>/apparat-cli/dist/pipelines/<name>/` into `<project>/.apparat/pipelines/<name>/`, refusing if the destination exists (no silent overwrite — operator owns reconciliation). Three lines of glob-and-copy reuse the same path resolver `pipeline-resolver.ts` already exposes. The two-tier architecture stops being implicit.
3. **Add `apparat pipeline watch <name> [--project x]`**. Long-running `chokidar`-style watcher on `<pipeline-folder>/**`; on every save runs `loadPipeline()` + `validateGraph()` and prints either `✓ valid (N nodes, M edges)` or the same `formatDiag` output the one-shot validator emits, plus the `diffEdgeLabels` already implemented (`pipeline/validate.ts:18-46`) but currently unwired to git history. Closes the silent edit loop and reuses every existing seam.
4. **Add `apparat pipeline rm <name> [--project x]`**. Deletes the folder, then asks the daemon socket to drop any heartbeat task whose pipeline path resolved to it. One verb, two surfaces, single transaction. Closes `D` and prevents schedule-vs-folder drift.
5. **Eat your own dogfood: bundle a `pipeline-create` *pipeline*.** A gate-driven interview (`type=gate` choices: agent / tool / gate; `inputs` for caller vars; loop / not loop), drafting `pipeline.dot` + agent stubs into `<project>/.apparat/pipelines/<name>/`, then routing into a validate node that re-runs until clean. The CLI command `apparat pipeline create <name>` becomes a 30-line shim that calls `pipelineRunCommand("pipeline-create", { variables: { target_name: name } })`. This is the deep-module collapse: one authoring seam, the meta-pipeline is the implementation. It also surfaces the `prompt-as-program` vs `engine-driven` story (`stimuli/prompt-as-program-philosophy.md`) — apparat is the engine pattern; using it to author its own graphs is the cleanest demonstration of why the engine pattern was worth building.
6. **Collapse the three authoring-reference channels.** README §"DOT file anatomy", `program.ts addHelpText` block, and `src/cli/skills/apparatus/pipelines.md` all describe the same primitives. Pick one source-of-truth (`pipelines.md` is already the "live reference" per ADR-0011) and have README + help text link to it instead of paraphrasing. Drift is silent today; collapse it before chunk-5's templates work returns.
