## Findings

1. **`2026-04-26-mark-plan-implemented-will-have-no-caller.md` has no frontmatter**
   - **What:** The plan file at `docs/superpowers/plans/2026-04-26-mark-plan-implemented-will-have-no-caller.md` begins with `# Mark Plan Implemented — Wire memory_writer As Canonical Caller — Implementation Plan` with no preceding `---` block. `list_plans(status=pending)` omits it; `list_plans(status=implemented)` omits it. The dispatched illumination `2026-04-26T0900-mark-plan-implemented-will-have-no-caller.md` links to it via `plan_path` — so the janitor's step-2 reconciliation loop reads the file, finds no `status: implemented`, and cannot close T0900.
   - **Evidence:** `list_plans(status=pending)` returns 7 files — this plan is absent. `read_file` of the plan returns content whose first line is `# Mark Plan Implemented…` with no frontmatter delimiter. T0900 illumination frontmatter: `plan_path: docs/superpowers/plans/2026-04-26-mark-plan-implemented-will-have-no-caller.md`.
   - **Why it matters:** T0900 will remain `dispatched` indefinitely — the janitor cannot close it even after the plan ships, because there is no `status: implemented` line for the reconciliation check to match. Mirrors the T1200 finding from T1100 exactly.
   - **Suggested action:** Prepend a two-line frontmatter block to the plan: `---\nstatus: pending\n---`. One-line fix; the janitor can then close T0900 on the run after the plan ships.

2. **`2026-04-26-runs-and-checkpoints-share-a-flat-namespace.md` has no frontmatter**
   - **What:** Same gap. `docs/superpowers/plans/2026-04-26-runs-and-checkpoints-share-a-flat-namespace.md` is also absent from `list_plans` in either status, and the dispatched illumination `2026-04-26T2000-runs-and-checkpoints-share-a-flat-namespace.md` links to it via `plan_path`.
   - **Evidence:** `list_plans(status=pending)` — 7 files returned, this plan absent. `read_file` returns content starting with `# Runs and Checkpoints Share a Flat Namespace — Implementation Plan`, no frontmatter delimiter.
   - **Why it matters:** T2000 has the same permanent-dispatch fate as T0900 above. Two dispatched illuminations from the same meditate session will accumulate without closure.
   - **Suggested action:** Same fix: prepend `---\nstatus: pending\n---`. Both plan files can be fixed in one pass.

3. **Root cause: plan-file generation produces no frontmatter**
   - **What:** Both plans were produced by the `illumination-to-plan.dot` pipeline (or an equivalent meditate session), yet neither carries lifecycle frontmatter. T1100 documented the same failure for plans generated before the lifecycle layer existed; today's occurrence proves the authoring path still does not inject frontmatter for new plans.
   - **Evidence:** The two oldest orphan plans in `list_plans(status=pending)` — `2026-04-12-headless-governance-gates.md`, `2026-04-12-meditate-backpressure-guard.md`, `2026-04-12-top-level-directory-map.md` — are also frontmatter-less pre-lifecycle plans, confirmed by T1100. Today's two plans repeat the pattern on post-lifecycle output.
   - **Why it matters:** Every plan the pipeline generates will be invisible to the janitor unless the authoring agent or the pipeline itself adds the frontmatter block. The janitor reconciliation loop has a structural blind spot at plan-creation time.
   - **Suggested action:** Add a `prepend_frontmatter` step to the `illumination-to-plan.dot` pipeline's `design_writer` or `plan_writer` node — or add a hard-rules bullet to the plan-writer agent rubric: "Every plan file must begin with `---\nstatus: pending\n---` before the `#` heading." This is a one-line agent rubric change; no engine edits required.
