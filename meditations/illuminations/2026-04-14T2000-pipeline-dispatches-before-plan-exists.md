## Core Idea

In `illumination-to-plan.dot`, the node order is `design_writer → mark_dispatched → plan_writer → done`. `mark_dispatched` is called with `plan_path: $design_doc_path` — the design document path — before the plan exists. The implementation plan written by `plan_writer` is never recorded anywhere: not in the illumination frontmatter, not in a registry, not in `IMPLEMENTATION_PLAN.md`. The pipeline's terminal deliverable lands in `docs/superpowers/plans/` with nothing pointing to it. The implement agent reads `IMPLEMENTATION_PLAN.md` and has no awareness that a plan was generated.

## Why It Matters

Three distinct failures compound from this single ordering error.

**Field name semantic inversion.** `markDispatched` stores `planPath` in frontmatter as a forward reference to the plan. Every caller of `list_illuminations(status=dispatched)` that reads `plan_path` expecting an implementation plan will find a design document instead. The field name promises one thing; the value delivers another. This is not a missing feature — it is an existing field with the wrong content.

**Premature state transition.** `mark_dispatched` fires after `design_writer` succeeds but before `plan_writer` runs. If `plan_writer` fails, the illumination is stuck in `dispatched` with a `plan_path` pointing to a design doc and no plan written. `markDispatched` requires `status=open` to proceed, so the illumination cannot be re-dispatched in a future pipeline run — it can only advance to `implemented` or `archived`. The pipeline has consumed the state token before earning it.

**Orphaned artifact.** `plan_writer` creates `docs/superpowers/plans/YYYY-MM-DD-<topic>.md` with no structured output (`plan_writer` has no `json_schema_file`). There is no `$plan_doc_path` variable. The plan path is not passed anywhere after `plan_writer` completes. The implement agent's prompt in `implement.md` says "Study IMPLEMENTATION_PLAN.md" — it reads a single manually-maintained root file, not `docs/superpowers/plans/`. Plans generated by the pipeline are invisible to the executor that would act on them. The graph's exit node `done` fires without the pipeline having bridged the design doc to the agent's working memory.

The agentic-loop-is-a-graph lens names the structural failure precisely: the pipeline has an explicit graph with named phases and transition conditions. But the terminal phase (`plan_writer`) produces an artifact that is not wired to any downstream consumer — the edge between the pipeline's output and the implement loop's input was never drawn.

## Revised Implementation Steps

1. **Add `json_schema_file` to `plan_writer`.** Create `pipelines/schemas/plan-writer.json` with a single required field `plan_doc_path: string`. Update the `plan_writer` node to reference it. This gives the pipeline a `$plan_doc_path` variable after `plan_writer` completes — without this, there is nothing to pass forward.

2. **Move `mark_dispatched` after `plan_writer`.** Reorder: `design_writer → plan_writer → mark_dispatched → done`. Update the `mark_dispatched` prompt to pass `plan_path: $plan_doc_path`. The illumination now records the actual implementation plan path, not the design doc path. The state transition fires only when both artifacts exist.

3. **Rename `design_doc_path` to make the two artifacts explicit.** The `mark_dispatched` prompt currently passes `plan_path: $design_doc_path`. After the reorder, pass both: `plan_path: $plan_doc_path`. Separately, update the `mark_dispatched` MCP tool signature (in `illumination-server.ts`) to add an optional `design_path` field — so dispatched illuminations can carry both artifact references. This is a backwards-compatible schema addition.

4. **Bridge plan output to `IMPLEMENTATION_PLAN.md`.** Add a `plan_injector` node after `plan_writer` and before `mark_dispatched`: `agent="implement"`, prompt: "Read the plan at `$plan_doc_path`. Append a new item to `IMPLEMENTATION_PLAN.md` linking to this plan with a one-line summary. Do not modify any other files." This closes the gap between the pipeline's output and the implement agent's working memory — the loop can see what the pipeline produced.

5. **Write a regression test for the ordering invariant.** In `src/cli/tests/pipeline-headless.test.ts`, add a test that mocks `mark_dispatched` and asserts it is called with a `plan_path` that points to a file in `docs/superpowers/plans/`, not `docs/superpowers/specs/`. Run the pipeline against a fixture illumination. This test documents the intended ordering and will fail if `mark_dispatched` is ever moved back above `plan_writer`.
