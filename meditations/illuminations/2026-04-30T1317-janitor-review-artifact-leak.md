## Findings

### 1. Plan-review artifacts appear in `list_plans` inventory as phantom plans

**What:** `docs/superpowers/plans/2026-04-30-produces-from-stdout-qualified-keys.review.md` is a review verdict file written by the plan-reviewer subagent during the `plan-writer` review loop. It has no YAML frontmatter (`status:`, `illumination_source:`), no chunk structure, and is not an implementation plan — yet `list_plans` (no filter) returns it in the inventory with the description `"Plan Review — produces_from_stdout qualified keys"`.

**Evidence:**
- `docs/superpowers/plans/2026-04-30-produces-from-stdout-qualified-keys.review.md:1`: `# Plan Review — produces_from_stdout qualified keys` — the file starts with a `# Plan Review` heading and `**Verdict:** Issues Found`, confirming it is a review artifact, not a plan.
- `list_plans` output (this run): `2026-04-30-produces-from-stdout-qualified-keys.review.md — Plan Review — produces_from_stdout qualified keys` — the MCP tool has no filter for `.review.md` suffixes; it globs all `.md` files in `docs/superpowers/plans/`.
- `list_plans status=pending` and `list_plans status=implemented`: the file is absent from both filtered results (no frontmatter → invisible to status filters). It occupies a third, unnamed state: visible to humans, invisible to lifecycle tools.

**Why it matters:** The plan-reviewer subagent (`general-purpose` agent dispatched via `Task` by `plan-writer`) writes its verdict to a file so the plan-writer can parse it. The natural path for that file is alongside the plan being reviewed, which places it in `docs/superpowers/plans/`. As more plans go through the review loop, the directory accumulates `.review.md` artifacts. The janitor's step 4 scan (`list_plans status=pending` — stale plan check) does not see them (no frontmatter), but users browsing the full `list_plans` output will see them interleaved with real plans, and any agent performing `glob_files("docs/superpowers/plans/*.md")` will enumerate them as peers.

**Suggested action (two options; pick one):**

*Option A — Filter at the `list_plans` level:* In `src/cli/mcp/illumination-server.ts`, change the `readdir` / `glob` call that backs `listPlans` to exclude files matching `*.review.md`. One-line pattern addition; preserves the ability to write reviews alongside plans without changing any other conventions.

*Option B — Redirect review artifacts to a subdirectory:* Change the plan-reviewer prompt template (in `superpowers:writing-plans`'s `plan-document-reviewer-prompt.md`) to write the review file to `docs/superpowers/plans/reviews/` instead of the plans directory root. Add `docs/superpowers/plans/reviews/` to `.gitignore` if reviews are meant to be ephemeral, or leave it tracked if reviews should be auditable alongside plans. This is a more structural fix that does not require changing the MCP server.

Either option must be paired with a `ralph pipeline validate` check (or a lint rule) that flags `.md` files in `docs/superpowers/plans/` that have no `status:` frontmatter — catching orphan plans (T1006) and review artifacts in the same pass.

### 2. `2026-04-30-produces-from-stdout-qualified-keys.md` plan also lacks frontmatter

**What:** The implementation plan for the `produces_from_stdout` qualified-keys work was generated this session — after `plan-writer.md` was updated to require `status: pending` + `illumination_source:` frontmatter (step 4) — but the plan file itself begins with `# Qualified Keys...` (no `---` block). It is absent from `list_plans status=pending` and its source illumination (`2026-04-30T0149-janitor-vision-tag-mismatch.md`, open) cannot be auto-closed by the janitor until the plan is marked implemented.

**Evidence:**
- `docs/superpowers/plans/2026-04-30-produces-from-stdout-qualified-keys.md:1`: `# Qualified Keys for \`produces_from_stdout\` Tool Outputs — Implementation Plan` — no preceding `---` block.
- `list_plans status=pending`: plan absent (no `status: pending` field to match).
- Plan contains `**Source illumination:** meditations/illuminations/2026-04-30T0149-janitor-vision-tag-mismatch.md` as a prose field, NOT as YAML frontmatter `illumination_source:` — prose is not machine-readable by the MCP server.

**Why it matters:** This is the T0421 orphan pattern repeating on the first plan written in the same session as the fix. Likely cause: the plan was generated by the same session that wrote the plan-writer fix — the two writes raced and the plan-writer emitted the plan before its own updated procedure was committed. The fix closes future plans; this one requires manual frontmatter backfill.

**Suggested action:** Prepend `---\nstatus: pending\nillumination_source: 2026-04-30T0149-janitor-vision-tag-mismatch.md\n---\n` to `docs/superpowers/plans/2026-04-30-produces-from-stdout-qualified-keys.md` as a standalone commit. After Chunk 2 of the plan ships, `memory-writer` can call `mark_plan_implemented` and the janitor can then close T0149.
